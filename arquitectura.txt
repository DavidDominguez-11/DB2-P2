# ParaMetrix — Arquitectura Frontend Completa

> Red social musical basada en Neo4j | Versión 1.0

---

## 1. STACK TECNOLÓGICO

| Capa | Tecnología | Justificación |
|------|-----------|---------------|
| Framework | React 18 + TypeScript | Componentes reutilizables, tipado fuerte |
| Estado servidor | TanStack Query (React Query) v5 | Cache, refetch, loading/error automático |
| Estado UI | Zustand | Liviano, para sidebar, modales, historial |
| HTTP Client | Axios | Interceptores, base URL, error handling |
| Routing | React Router v6 | Rutas anidadas, lazy loading |
| Estilos | Tailwind CSS + CSS Variables | Utilidades + tema consistente |
| Formularios | React Hook Form + Zod | Validación declarativa y tipada |
| Tablas | TanStack Table v8 | Paginación, filtros, sort reutilizable |
| Gráficas | Recharts | Analytics charts |
| Notificaciones | Sonner | Toast feedback |
| Editor código | CodeMirror 6 | Cypher Console con syntax highlight |

### client.ts
```typescript
const api = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  headers: { 'Content-Type': 'application/json' }
});
// Interceptor: manejo global de errores, toast automático
```

### endpoints.ts — Todas las URLs disponibles


## 4. PANTALLAS — DESCRIPCIÓN DETALLADA

---

### 4.1 HOME — Feed Social

**Ruta:** `/`

**Descripción:** Feed cronológico de posts de usuarios. Muestra interacciones sociales (likes, escuchas). Panel lateral con recomendaciones rápidas.

**Componentes:**
- `PostCard` — muestra caption, tipo, fecha, hashtags, privacidad
- `CreatePostModal` — formulario validado con React Hook Form
- `InteractionButtons` — botones Like / Share / Listen
- Paginación infinite scroll con React Query `useInfiniteQuery`

**Acciones → Endpoints:**

| Acción | Método | Endpoint |
|--------|--------|----------|
| Cargar feed | GET | `/api/v1/posts/?skip=0&limit=25` |
| Crear post | POST | `/api/v1/posts/` |
| Editar post | PATCH | `/api/v1/posts/{post_id}` |
| Eliminar post | DELETE | `/api/v1/posts/{post_id}` |
| Like post (relación) | POST | `/api/v1/interactions/` |

**Validaciones CreatePost:**
- `post_id`: requerido, string único (generado con UUID en frontend)
- `caption`: requerido, mínimo 10 caracteres
- `fecha`: requerido, formato date ISO
- `tipo`: requerido, enum [song, playlist, update, event]
- `privacidad`: default "public"

---

### 4.2 PROFILE — Perfil de Usuario

**Ruta:** `/profile/:userId`

**Descripción:** Vista de perfil con información personal, estadísticas de actividad, seguidores y canciones escuchadas. Permite editar el propio perfil.

**Componentes:**
- `ProfileHeader` — avatar generado, username, email, premium badge
- `ActivityFeed` — últimos posts del usuario
- `FollowerList` — tabla de usuarios relacionados
- `EditProfileModal` — PATCH usuario

**Acciones → Endpoints:**

| Acción | Método | Endpoint |
|--------|--------|----------|
| Obtener usuario | GET | `/api/v1/users/{user_id}` |
| Editar perfil | PATCH | `/api/v1/users/{user_id}` |
| Posts del usuario | GET | `/api/v1/posts/?user_id=...` |
| Seguidores (relaciones) | GET | `/api/v1/interactions/?to_id=...&rel_type=FOLLOWS` |
| Canciones escuchadas | GET | `/api/v1/interactions/?from_id=...&rel_type=LISTENED` |

---

### 4.3 EXPLORE — Exploración

**Ruta:** `/explore`

**Descripción:** Buscador y explorador de canciones, géneros y usuarios. Tabs por entidad. Paginación real con skip/limit.

**Tabs:**
- 🎵 Canciones → `SongGrid` 
- 🎸 Géneros → `GenreList`
- 👤 Usuarios → `UserSearchList`

**Acciones → Endpoints:**

| Acción | Método | Endpoint |
|--------|--------|----------|
| Listar canciones | GET | `/api/v1/songs/?skip=N&limit=25` |
| Detalle canción | GET | `/api/v1/songs/{song_id}` |
| Listar géneros | GET | `/api/v1/genres/?skip=N&limit=25` |
| Listar usuarios | GET | `/api/v1/users/?skip=N&limit=25` |
| Filtrar premium | GET | `/api/v1/users/?premium=true` |

---

### 4.4 PLAYLISTS — Gestión

**Ruta:** `/playlists`

**Descripción:** CRUD completo de playlists. Vista de detalle con canciones asociadas. Botón para agregar canciones via relación.

**Acciones → Endpoints:**

| Acción | Método | Endpoint |
|--------|--------|----------|
| Listar playlists | GET | `/api/v1/playlists/` |
| Crear playlist | POST | `/api/v1/playlists/` |
| Editar playlist | PATCH | `/api/v1/playlists/{playlist_id}` |
| Eliminar playlist | DELETE | `/api/v1/playlists/{playlist_id}` |
| Añadir canción (rel) | POST | `/api/v1/interactions/` (CONTAINS) |

**Validaciones CreatePlaylist:**
- `playlist_id`, `nombre`, `descripcion`, `fecha_creacion`: requeridos
- `nombre`: min 3, max 100 caracteres
- `fecha_creacion`: formato date

---

### 4.5 ANALYTICS — Dashboard

**Ruta:** `/analytics`

**Descripción:** Dashboard de data science con métricas, recomendaciones y visualizaciones de grafos.

**Sub-paneles:**

#### OverviewPanel
- Stats cards: total usuarios, canciones, playlists, géneros
- Fuente: agregaciones COUNT de cada entidad

#### RecommendationsPanel
- Recomendaciones de canciones basadas en Jaccard similarity
- ⚠️ **BACKEND REQUERIDO:** `GET /api/v1/analytics/recommendations/{user_id}`
  - Lógica: usuarios con gustos similares → canciones no escuchadas

#### SimilarUsersPanel
- Usuarios con géneros favoritos en común
- ⚠️ **BACKEND REQUERIDO:** `GET /api/v1/analytics/similar-users/{user_id}`

#### InfluenceRanking
- Top usuarios por número de seguidores
- **Implementado via Cypher Console:**
  ```cypher
  MATCH (u:User)<-[:FOLLOWS]-(f) 
  RETURN u.username, count(f) as seguidores 
  ORDER BY seguidores DESC LIMIT 10
  ```

#### GenreDistribution
- Distribución de géneros por popularidad
- Fuente: `GET /api/v1/genres/aggregate?group_by=nombre&agg_field=popularidad&agg_func=sum`

**Acciones → Endpoints:**

| Acción | Método | Endpoint |
|--------|--------|----------|
| Agg usuarios | GET | `/api/v1/users/aggregate?group_by=premium&agg_func=count` |
| Agg canciones | GET | `/api/v1/songs/aggregate?group_by=popularidad&agg_func=avg` |
| Agg géneros | GET | `/api/v1/genres/aggregate?group_by=nombre&agg_field=popularidad&agg_func=sum` |
| Agg playlists | GET | `/api/v1/playlists/aggregate` |

## 5. MÓDULO GRAPH MANAGER — Diseño Detallado

**Ruta base:** `/graph-manager`

**Navegación interna:** Tabs laterales entre sub-módulos

---

### 5.1 NODE BUILDER

**Ruta:** `/graph-manager/node-builder`

**Descripción:** Formulario para crear nodos en Neo4j. Soporta modo single-label (endpoints de entidad) y modo multi-label (endpoint `/api/v1/nodes`).

#### Toggle: Single Label vs Multi-Label

```
[Single Label ●────] [Multi Label ────●]
```

#### Modo Single Label

Selector de entidad: User | Song | Playlist | Post | Genre

Formulario para **User** (mínimo 5 propiedades):
- `user_id` — string, requerido, único
- `username` — string, requerido, min 3
- `email` — string, formato email, requerido
- `fecha_registro` — date, requerido
- `premium` — boolean, default false
- `generos_favoritos` — array de strings
- `is_artist` — boolean, default false

→ `POST /api/v1/users/`

Formulario para **Song** (mínimo 5 propiedades):
- `song_id`, `titulo`, `duracion`, `fecha_lanzamiento`, `popularidad`, `idiomas`

→ `POST /api/v1/songs/`

Formulario para **Genre** (mínimo 5 propiedades):
- `genre_id`, `nombre`, `descripcion`, `popularidad`, `origen`, `activo`

→ `POST /api/v1/genres/`

Formulario para **Playlist** (mínimo 5 propiedades):
- `playlist_id`, `nombre`, `descripcion`, `fecha_creacion`, `publica`, `numero_canciones`, `is_featured`

→ `POST /api/v1/playlists/`

Formulario para **Post** (mínimo 5 propiedades):
- `post_id`, `caption`, `fecha`, `tipo`, `privacidad`, `hashtags`, `is_ad`

→ `POST /api/v1/posts/`

#### Modo Multi-Label

Labels selector (checkboxes múltiples):
```
☑ User   ☑ Artist   ☐ Influencer   ☐ Moderator   ☐ Label
```

Sección de propiedades dinámica:
- Campo key-value dinámico (useFieldArray)
- Botón "+ Agregar propiedad"
- Mínimo 5 propiedades requeridas (validación Zod)
- Tipos soportados: string, number, boolean, date

→ `POST /api/v1/nodes` con body `{ labels: [...], properties: {...} }`

**Extensión labels:**
- `PATCH /api/v1/nodes/{id}/labels` para agregar/quitar labels post-creación
- `GET /api/v1/nodes/{id}/labels` para ver labels actuales

**Validaciones:**
- ✔ Mínimo 1 label seleccionado
- ✔ Mínimo 5 propiedades en modo multi-label
- ✔ Keys de propiedades sin espacios, sin caracteres especiales
- ✔ ID único requerido en todos los modos

---

### 5.2 NODE EXPLORER

**Ruta:** `/graph-manager/node-explorer`

**Descripción:** Consulta de nodos individuales o en masa. Incluye panel de agregaciones.

#### Tab: Nodo Individual
- Input: Tipo de entidad (selector) + ID
- Botón "Consultar"
- Resultado: `NodeCard` con todas las propiedades
- → `GET /api/v1/{entity}/{id}`

#### Tab: Múltiples Nodos
- Selector de entidad
- Controles de paginación (skip/limit)
- Filtros disponibles: `premium` (users)
- Tabla con columnas dinámicas según entidad
- → `GET /api/v1/{entity}/?skip=N&limit=M`

#### Tab: Agregaciones
Formulario:
- Entidad: User | Song | Playlist | Post | Genre
- `group_by`: campo por el que agrupar
- `agg_field`: campo a agregar
- `agg_func`: count | avg | sum | min | max

Resultado: tabla + bar chart

→ `GET /api/v1/{entity}/aggregate?group_by=X&agg_field=Y&agg_func=Z`

**Endpoints:**
```
GET /api/v1/users/aggregate
GET /api/v1/songs/aggregate
GET /api/v1/playlists/aggregate
GET /api/v1/posts/aggregate
GET /api/v1/genres/aggregate
```

---

### 5.3 PROPERTY MANAGER

**Ruta:** `/graph-manager/property-manager`

**Descripción:** Actualización y eliminación de propiedades de nodos, tanto individual como masivamente.

#### Tab: Operaciones Individuales

Modo **Actualizar propiedad:**
- Selector entidad + ID
- Campo: nombre de propiedad
- Nuevo valor
- → `PATCH /api/v1/{entity}/{id}` con body `{ campo: valor }`

Modo **Eliminar propiedades:**
- Selector entidad + ID
- Multi-select de propiedades a eliminar
- → `DELETE /api/v1/{entity}/{id}/properties?properties=prop1&properties=prop2`

#### Tab: Operaciones Masivas

**Bulk Update Nodos:**
```json
{
  "label": "User",
  "filter_property": "premium",
  "filter_value": false,
  "update_data": { "is_new_user": true }
}
```
→ `PATCH /api/v1/bulk/nodes`

**Bulk Remove Property de Nodos:**
```json
{
  "label": "Song",
  "property_to_remove": "temp_field",
  "filter_property": "popularidad",
  "filter_value": 0
}
```
→ `DELETE /api/v1/bulk/nodes/properties`

---

### 5.4 RELATIONSHIP BUILDER

**Ruta:** `/graph-manager/relationship-builder`

**Descripción:** Creación de relaciones entre dos nodos. Selector visual de origen y destino. Mínimo 3 propiedades de relación.

**Campos del formulario:**
- Nodo origen: label + ID (selector con búsqueda)
- Nodo destino: label + ID (selector con búsqueda)
- Tipo de relación: LISTENED | FOLLOWS | LIKED | CONTAINS | POSTED | etc.
- Propiedades (mínimo 3, campo dinámico):
  - Ejemplos: `fecha`, `duracion_escucha`, `plataforma`, `dispositivo`, `contexto`

→ `POST /api/v1/interactions/`

**Body:**
```json
{
  "from_label": "User",
  "from_id": "user-123",
  "to_label": "Song",
  "to_id": "song-456",
  "rel_type": "LISTENED",
  "properties": {
    "fecha": "2024-01-15",
    "duracion_escucha": 245,
    "plataforma": "web"
  }
}
```

**Validaciones:**
- ✔ Origen y destino no pueden ser el mismo ID
- ✔ Mínimo 3 propiedades en la relación
- ✔ `rel_type` no puede estar vacío
- ✔ IDs de nodos deben existir (verificación opcional via GET)

**Tipos de relación disponibles (predefinidos + campo libre):**
```
LISTENED | FOLLOWS | LIKED | CONTAINS | POSTED | 
BELONGS_TO | FEATURED_IN | COLLABORATED_WITH
```

---

### 5.5 RELATIONSHIP MANAGER

**Ruta:** `/graph-manager/relationship-manager`

**Descripción:** Editar y eliminar relaciones, individual y masivamente.

#### Tab: Operaciones Individuales

**Editar relación:**
- Input: ID de interacción
- `GET /api/v1/interactions/{id}` para obtener propiedades actuales
- Formulario editable
- → `PATCH /api/v1/interactions/{id}` con `{ properties: {...} }`

**Eliminar propiedades de relación:**
- Input: ID de interacción
- Multi-select de propiedades
- → `DELETE /api/v1/interactions/{id}/properties` con `{ property_names: [...] }`

**Eliminar relación:**
- Input: ID de interacción
- Confirmación
- → `DELETE /api/v1/interactions/{id}`

#### Tab: Operaciones Masivas

**Bulk Update Relaciones:**
```json
{
  "rel_type": "LISTENED",
  "filter_property": "plataforma",
  "filter_value": "mobile",
  "update_data": { "migrated": true }
}
```
→ `PATCH /api/v1/bulk/relationships`

**Bulk Delete Relaciones:**
```json
{
  "rel_type": "LISTENED",
  "filter_property": "fecha",
  "filter_value": "2020-01-01"
}
```
→ `DELETE /api/v1/bulk/relationships`

**Bulk Remove Property de Relaciones:**
```json
{
  "rel_type": "FOLLOWS",
  "property_to_remove": "temp_field"
}
```
→ `DELETE /api/v1/bulk/relationships/properties`

### 5.7 CYPHER CONSOLE

**Ruta:** `/graph-manager/cypher-console`

**Descripción:** Terminal interactiva para ejecutar consultas Cypher directamente contra Neo4j via API.

**Layout:**
```
┌─────────────────────────────────┬──────────────────┐
│  Editor (CodeMirror)            │  Historial        │
│  MATCH (n:User)                 │  > Query 1        │
│  RETURN n LIMIT 10              │  > Query 2        │
│                    [Ejecutar ▶] │  > Query 3        │
├─────────────────────────────────┴──────────────────┤
│  Resultados                                         │
│  ┌──────┬──────────┬───────────┬──────────────┐    │
│  │ id   │ username │ email     │ premium      │    │
│  ├──────┼──────────┼───────────┼──────────────┤    │
│  │ u1   │ juanito  │ j@m.com   │ false        │    │
└─────────────────────────────────────────────────────┘
```

**Endpoint usado:**
→ `POST /api/v1/cypher/query` con `{ "query": "..." }`

**Queries predefinidas (Templates):**

```cypher
-- Query 1: Usuarios más activos
MATCH (u:User)-[r:LISTENED]->()
RETURN u.username, count(r) as escuchas
ORDER BY escuchas DESC LIMIT 10

-- Query 2: Canciones más populares
MATCH (s:Song)<-[:LISTENED]-(u:User)
RETURN s.titulo, count(u) as oyentes
ORDER BY oyentes DESC LIMIT 10

-- Query 3: Ranking de influencia (seguidores)
MATCH (u:User)<-[:FOLLOWS]-(f:User)
RETURN u.username, count(f) as seguidores
ORDER BY seguidores DESC LIMIT 10

-- Query 4: Usuarios con géneros en común
MATCH (u1:User)-[:LIKED]->(g:Genre)<-[:LIKED]-(u2:User)
WHERE u1 <> u2
RETURN u1.username, u2.username, collect(g.nombre) as generos_comunes
LIMIT 20

-- Query 5: Playlists con más canciones
MATCH (p:Playlist)-[:CONTAINS]->(s:Song)
RETURN p.nombre, count(s) as total_canciones
ORDER BY total_canciones DESC LIMIT 10

-- Query 6: Nodos aislados (sin relaciones)
MATCH (n)
WHERE NOT (n)--()
RETURN labels(n), n LIMIT 20
```

**Funcionalidades:**
- Historial persistido en Zustand + localStorage
- Botón "Copiar query"
- Botón "Cargar template"
- Exportar resultados como CSV
- Manejo de errores Cypher (mensaje legible)
- Límite de 1000 resultados en frontend para performance

```typescript
// hooks/useUsers.ts
export const useUsers = (params: UserListParams) =>
  useQuery({
    queryKey: ['users', params],

  });

export const useCreateUser = () =>
  useMutation({
    mutationFn: usersApi.create,
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ['users'] });
      toast.success('Usuario creado exitosamente');
    },
    onError: (e: AxiosError) => toast.error(e.message),
  });
```

### Patrón de Optimistic Updates (para operaciones frecuentes)
```typescript
export const useDeleteNode = (entity: string) =>
  useMutation({
    mutationFn: (id: string) => entityApi(entity).delete(id),
    onMutate: async (id) => {
      await queryClient.cancelQueries({ queryKey: [entity] });
      const prev = queryClient.getQueryData([entity]);
      queryClient.setQueryData([entity], (old: any[]) =>
        old.filter(n => n.id !== id)
      );
      return { prev };
    },
    onError: (_err, _id, ctx) => {
      queryClient.setQueryData([entity], ctx?.prev);
    },
  });
```
